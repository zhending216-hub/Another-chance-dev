这样你看完会更清楚：用户点一次“续写故事”，系统内部到底做了哪些事，哪些能力已经接上了，哪些还只是设计了但没完全打通。\n\n **总览** \n- 这套系统的核心不是“一次把整本故事生成完”，而是“以段落为单位，沿着某个分支持续生成下一段”。\n- 整个链路可以概括成 8 步：创建故事 → 读取当前分支上下文 → 构造 Prompt → 调 AI → 流式回传文本 → 保存新段落 → 异步补摘要/配图 → 用户继续续写或从某段分叉。\n- 真正决定生成质量的核心在 3 个地方：上下文链怎么取、Prompt 怎么拼、生成后怎样把结果重新写回故事树。\n\n **1. 数据底座先搭好** \n- 所有运行时数据都不是进数据库，而是直接落在 `data/` 下的 JSON 文件里，统一由 `SimpleStore` 管理，定义在 `src/lib/simple-db.ts:15`。\n- 故事、段落、分支分别存在 `stories.json`、`segments.json`、`branches.json`，对应结构见 `src/lib/simple-db.ts:38`、`src/lib/simple-db.ts:52`、`src/lib/simple-db.ts:73`。\n- 最关键的方法是 `getOrderedChain`，它负责“把当前分支真正应该看到的上下文串起来”，实现位于 `src/lib/simple-db.ts:90`。\n- 主线分支很简单：从根段落开始，顺着 `parentSegmentId` 一直往后找。\n- 非主线分支会先拼上“主线里直到分叉点为止的上下文”，再接上这个分支自己的段落，所以 AI 在分支里续写时并不会丢失主线前史，逻辑在 `src/lib/simple-db.ts:106`。\n\n **2. 创建故事时，实际发生了什么** \n- 前端创建页会把标题、描述、storyType、era，甚至 `prompt` 和初始角色一起发给后端，发起位置在 `src/app/create/page.tsx:104` 和 `src/app/create/page.tsx:133`。\n- 但后端真正读取的只有 `title / description / author / genre / era / storyType`，代码在 `src/app/api/stories/route.ts:31`。\n- 后端会先创建一条 Story 记录，然后立刻给它插入一个“开篇段落”，但这个开篇不是 AI 写的，而只是占位文本：`《xxx》的故事开始了...`，见 `src/app/api/stories/route.ts:71`。\n- 也就是说，当前实现里“创建故事”这一步并没有真正调用大模型生成开篇；真正的 AI 创作，是从你后面点“继续续写”或“创建分叉”开始的。\n- 这也带来一个副作用：第一次续写时，系统的“风格锚点”其实只能参考这句占位文本，风格约束会比较弱。\n\n **3. 用户点“继续续写”后，前端怎么触发** \n- 故事详情页加载时，会把故事本体、树结构、当前分支段落都取回来，相关逻辑在 `src/app/story/[id]/page.tsx:74`、`src/app/story/[id]/page.tsx:82`、`src/app/story/[id]/page.tsx:90`。\n- 用户点击“继续”时，前端调用的是流式接口 `/api/stories/[id]/stream-continue`，入口在 `src/app/story/[id]/page.tsx:173`。\n- 请求里会带两个关键参数：当前分支 `branchId`，以及节奏配置 `pacingConfig`，见 `src/app/story/[id]/page.tsx:181`。\n- 默认 UI 走的是“流式续写”这条链，而不是非流式接口。\n\n **4. 后端收到续写请求后，先做上下文准备** \n- 流式续写的服务端入口在 `src/app/api/stories/[id]/stream-continue/route.ts:16`。\n- 它先校验故事是否存在，再根据 `storyId + branchId` 用 `getOrderedChain` 取出完整上下文链，关键代码在 `src/app/api/stories/[id]/stream-continue/route.ts:25` 和 `src/app/api/stories/[id]/stream-continue/route.ts:29`。\n- 然后拿链尾段落 `tailSegment` 作为“新段落的父节点”，后面保存时会把新段落挂到它后面，见 `src/app/api/stories/[id]/stream-continue/route.ts:33`。\n- 在真正生成前，还会先做一次“前文矛盾检测”，避免已有剧情自相矛盾，逻辑在 `src/app/api/stories/[id]/stream-continue/route.ts:60`。\n- 这里要注意：流式接口会做一致性检测，但默认路径里并不会像非流式接口那样先跑时间轴校验；时间轴校验只在 `src/app/api/stories/[id]/continue/route.ts:44` 这条非流式链里明确执行。\n\n **5. Prompt 是整个系统最核心的一步** \n- Prompt 统一由 `buildFullPrompt` 生成，定义在 `src/lib/prompt-builder.ts:111`。\n- 这个函数不是简单把前文拼起来，而是按一套固定顺序把很多“约束层”叠进去，顺序注释就写在 `src/lib/prompt-builder.ts:1`。\n- 先做体裁判断：优先用故事本身的 `genre`，没有就从描述里猜，判断逻辑在 `src/lib/prompt-builder.ts:122`，配置来源是 `src/lib/genre-config.ts:8`。\n- 再根据体裁决定写作风格：历史题材偏半文半白，科幻/悬疑/都市/同人偏现代白话文，见 `src/lib/prompt-builder.ts:151`。\n- 如果系统判断这是虚构作品，但最近两段明显充满古风虚词，它还会强制加入“风格覆盖指令”，要求 AI 立刻改用现代白话文，逻辑在 `src/lib/prompt-builder.ts:171`。\n- 然后补上故事标题、背景，以及“风格锚点”——取第一段前 200 字作为模仿样例，见 `src/lib/prompt-builder.ts:191` 和 `src/lib/prompt-builder.ts:196`。\n\n **6. Prompt 里会塞哪些上下文模块** \n- 角色信息：如果故事或段落上挂了 `characterIds`，就会把角色的性格、动机、口癖、当前状态、关系网转成 Prompt，逻辑在 `src/lib/prompt-builder.ts:235`，生成格式在 `src/lib/character-engine.ts:135`。\n- 活跃事件：理论上会把未解决冲突、近期解决事件塞进去，逻辑在 `src/lib/prompt-builder.ts:213` 和 `src/lib/event-tracker.ts:176`。\n- 分支记忆：如果当前故事已经出现多个分支，会把“其他分支发生了什么”摘要给模型，逻辑在 `src/lib/prompt-builder.ts:223` 和 `src/lib/branch-memory.ts:149`。\n- 导演模式：如果用户手动给了角色状态、世界变量、硬约束，这部分会优先插在前文之前，见 `src/lib/prompt-builder.ts:247`，数据来源是 `src/lib/director-manager.ts:69`。\n- 前文正文：不是把所有旧段落原封不动塞进去，而是做“近段全文 + 远段摘要”的分层压缩，逻辑在 `src/lib/prompt-builder.ts:268` 和 `src/lib/context-summarizer.ts:313`。\n- 时间线/世界观：历史题材会把时间轴和对应朝代的 lorebook 设定塞进去，逻辑在 `src/lib/prompt-builder.ts:302`，格式化在 `src/lib/timeline-engine.ts:119`，lorebook 来源是 `src/lib/lorebook.ts:17`。\n- 节奏控制：把 `rush / detailed / pause / summary` 转成字数和写作风格要求，逻辑在 `src/lib/prompt-builder.ts:338`，规则在 `src/lib/pacing-engine.ts:8`。\n- 最后再补“硬约束”和“请续写下一段”的最终指令，见 `src/lib/prompt-builder.ts:344` 和 `src/lib/prompt-builder.ts:362`。\n\n **7. 上下文压缩是怎么做的** \n- 这是这个项目里比较值钱的一层设计。\n- 最近 5 段保留全文，更早的段落先变成段落摘要，再合并成组摘要，必要时再压缩成更粗的章级摘要，逻辑在 `src/lib/context-summarizer.ts:313`。\n- 真正给 Prompt 用的文本由 `getContextForPrompt` 产出，见 `src/lib/context-summarizer.ts:474`。\n- 它会把“远端前文概要”和“近期完整内容”分开，尽量让模型既知道长程剧情，又不浪费上下文窗口。\n- 新段落保存后，系统还会异步预生成该段的摘要缓存，下一次续写时就能直接复用，触发点在 `src/app/api/stories/[id]/stream-continue/route.ts:200`。\n\n **8. 真正调用 AI 时做了什么** \n- 非流式调用统一走 `ai-client`，支持优先级队列、并发控制、每分钟限流、429/5xx 指数退避重试，核心在 `src/lib/ai-client.ts:60`、`src/lib/ai-client.ts:200`、`src/lib/ai-client.ts:381`。\n- 同时它会根据故事体裁调 `temperature`：历史类更保守，虚构类更开放，逻辑在 `src/lib/ai-client.ts:310`。\n- 但默认前端走的“流式续写”有个细节：它没有直接调用 `callAI`，而是自己用 `buildOpenAIRequest` 拼完请求后 `fetch`，代码在 `src/app/api/stories/[id]/stream-continue/route.ts:84`。\n- 这意味着：流式主生成链路会复用参数配置，但不会自动享受到 `ai-client` 里的队列和重试封装。\n- 也就是说，主文本的流式生成更直接，但容错链路比非流式略薄一些。\n\n **9. 流式文本返回到前端的过程** \n- 后端把上游模型流式响应解析成 SSE，再转发给前端，核心循环在 `src/app/api/stories/[id]/stream-continue/route.ts:119`。\n- 如果配置了节奏控制，后端会尽量按“完整句子/换行”切成 `line` 事件；超过每步行数上限时，会发一个 `pause` 事件，见 `src/app/api/stories/[id]/stream-continue/route.ts:140`。\n- 前端收到这些 SSE 片段后，会把内容不断拼成 `newContent`，并按 `maxLinesPerStep` 控制当前显示几行，逻辑在 `src/app/story/[id]/page.tsx:145` 和 `src/app/story/[id]/page.tsx:196`。\n- 所以用户看到的“慢慢出来的故事”，本质上是：后端从大模型 token 流里持续抽取内容，前端再做一层按行展示。\n\n **10. 生成完成后，系统如何落盘** \n- 当流式响应结束后，后端会把累计出来的 `fullContent` 组装成一个新的 `StorySegment`，并写入 `segments.json`，逻辑在 `src/app/api/stories/[id]/stream-continue/route.ts:171`。\n- 这个新段落会带上 `branchId`、`parentSegmentId`、`narrativePace`、`mood` 等元数据，后面再读取分支链时就能接上。\n- 保存成功后，前端重新拉一遍当前分支段落和树结构，界面上的故事就“长出新的一段”了，见 `src/app/story/[id]/page.tsx:225`。\n\n **11. 生成后还有两个异步后处理** \n- 第一是摘要缓存：系统后台异步为新段落生成摘要，给下一次 Prompt 压缩上下文用，见 `src/app/api/stories/[id]/stream-continue/route.ts:200`。\n- 第二是配图：它会把图片生成作为低优先级任务塞进 AI 请求队列，生成完成后回写到段落的 `imageUrls`，见 `src/app/api/stories/[id]/stream-continue/route.ts:204`。\n- 所以文字生成是主流程，摘要和插图都是“尾处理”，不会阻塞用户先看到文字。\n\n **12. 分叉生成和普通续写有什么不同** \n- 用户点击某段的“分叉”时，前端会先让你选方向，然后调用 `/api/stories/[id]/branch`，触发代码在 `src/app/story/[id]/page.tsx:270` 和 `src/app/story/[id]/page.tsx:302`。\n- 后端先创建一个 branch 记录，并把当前段标记为分叉点，逻辑在 `src/app/api/stories/[id]/branch/route.ts:35` 和 `src/app/api/stories/[id]/branch/route.ts:51`。\n- 接着它只取“主线里直到这个分叉点为止”的上下文，再用同一个 `buildFullPrompt` 生成 Prompt，见 `src/app/api/stories/[id]/branch/route.ts:55` 和 `src/app/api/stories/[id]/branch/route.ts:71`。\n- 最后在 Prompt 尾部额外追加一句“用户希望的故事走向：xxx”，强行把模型往这个方向带，见 `src/app/api/stories/[id]/branch/route.ts:82`。\n- AI 回来后，会直接创建这个新分支的第一段，并把它挂在分叉源段落下面，见 `src/app/api/stories/[id]/branch/route.ts:96`。\n\n **13. 一致性控制具体做了哪些事** \n- 一致性检测由 `ConsistencyChecker` 完成，入口在 `src/lib/consistency-checker.ts:70`。\n- 它会检查三类问题：角色矛盾、时间矛盾、世界观矛盾。\n- 角色层面会查“已死亡角色又活蹦乱跳出现”“性格突然反转”“关系突然矛盾”，逻辑从 `src/lib/consistency-checker.ts:72` 开始。\n- 时间层面会查季节不匹配、年份回退、朝代穿帮，逻辑从 `src/lib/consistency-checker.ts:166` 开始。\n- 时间轴单调性校验则是另一套 `TimelineEngine`，会查年份和季节是否倒退，见 `src/lib/timeline-engine.ts:28`。\n- 非流式续写会在生成前显式做时间轴校验；流式续写默认只做一致性检测，不会把时间轴违规作为单独预警抛出来。\n\n **14. 你可以把它理解成一条“闭环”** \n- 不是“AI 一次写完故事”，而是“故事状态机”每次向前推进一步。\n- 每推进一次，系统都会更新三类状态：文本状态（segments）、结构状态（branches/tree）、辅助记忆状态（summaries/images/可能的角色与导演状态）。\n- 下一次生成再把这些状态重新读回来，重新组 Prompt，再生成下一段。\n- 所以这其实是一个“增量生成 + 增量记忆 + 可分叉结构化存档”的系统，不是普通聊天式写作。\n\n **15. 当前实现里，最值得注意的几个现实差异** \n- `创建故事` 这一步并没有真正 AI 生成开篇，只是插入占位段落，见 `src/app/api/stories/route.ts:71`。\n- 创建页传了 `prompt` 和 `characters`，但
