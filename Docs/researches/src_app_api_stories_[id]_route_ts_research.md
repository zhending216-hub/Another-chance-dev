# 研究报告: src/app/api/stories/[id]/route.ts

## 概述

该文件实现了单个故事的查询 API，是故事详情页的数据基础接口。通过故事 ID 获取故事的完整元数据。

## 用途

前端故事阅读页面加载时，首先调用此接口获取故事基本信息（标题、描述、作者、创建时间等），然后再加载段落和分支数据。

## 导出

- `GET` 函数：接收路径参数 `id`，返回 `{ success, story }` 或 404 错误。

## 依赖

- `@/lib/simple-db`：`storiesStore`。

## 核心逻辑

1. 从 `storiesStore` 加载所有故事。
2. 通过 `find` 方法按 ID 查找目标故事。
3. 找到则返回完整故事对象，未找到则返回 404。

逻辑极其简单，是典型的 CRUD 中的 Read 操作。

## 数据流

```
GET /api/stories/[id] → storiesStore.load() → find by id → return story
```

## ChronosMirror 升级改进点

### 1. 角色建模
- 故事元数据应包含角色列表和角色关系信息。
- 可在 GET 响应中附加角色摘要（角色名、阵营、当前状态）。
- 故事对象应支持"角色面板"数据结构。

### 2. 时间轴校验
- 故事应包含时间范围信息（起止年份、朝代）。
- 可添加 `era`、`startYear`、`endYear` 等字段。
- 时间轴校验在创建故事时就应该开始——确保标题与朝代匹配。

### 3. MCP 维基百科
- 获取故事详情时，可通过 MCP 查询维基百科获取该历史事件的概述，作为故事元数据的补充。
- 可自动生成"历史背景"摘要。

### 4. 节奏控制
- 可返回故事的阅读进度信息（当前阅读到哪个段落、已读分支数等）。
- 可添加"推荐阅读路径"，引导用户体验最佳叙事顺序。
