# 研究报告: src/app/api/stories/[id]/branch/route.ts

## 概述

该文件实现了故事分支创建的核心 API。当用户在故事阅读过程中选择"从此处分叉"时，前端调用此 POST 接口，系统会基于用户选择的方向，从当前段落创建一条新的故事分支线，并通过 AI 生成该分支的首个段落内容。

## 用途

实现"历史分叉"这一核心玩法。用户可以在故事任意节点选择不同的历史走向（如"荆轲成功刺杀秦王"），系统创建分支记录，AI 生成符合该走向的续写内容。这是古事平台区别于普通故事生成器的关键特性——多时间线叙事。

## 导出

- `POST` 函数：接收 `{ segmentId, userDirection, branchTitle }` 请求体，返回新创建的分支和首个段落。

## 依赖

- `@/lib/simple-db`：使用 `storiesStore`、`segmentsStore`、`branchesStore`、`getOrderedChain`，以及 `StorySegment` 类型。
- AI API：通过 `fetch` 调用 `AI_BASE_URL/chat/completions`，使用 `AI_MODEL` 和 `AI_API_KEY` 环境变量。

## 核心逻辑

1. **参数验证**：要求 `storyId`、`segmentId`、`userDirection` 三个必填参数。
2. **存在性检查**：验证故事和源段落都存在。
3. **创建分支记录**：生成唯一 `branchId`，记录源段落、用户方向、时间戳等信息，保存到 `branchesStore`。
4. **标记分叉点**：将源段落的 `isBranchPoint` 设为 `true`。
5. **构建上下文**：从主分支获取到源段落的有序链作为 AI 上下文（通过 `getOrderedChain` + `slice`）。
6. **AI 生成**：构建包含故事标题、背景、前文进展和用户方向的 prompt，调用 AI 生成 150-300 字的续写内容。
7. **创建新段落**：将 AI 生成的内容作为新分支的首个段落，设置 `parentSegmentId` 为源段落、`branchId` 为新分支 ID。
8. **返回结果**：同时返回分支记录和新段落。

## 数据流

```
用户点击"分叉" → POST /api/stories/[id]/branch
→ 验证参数 → 创建 branch 记录 → 标记 isBranchPoint
→ 获取主链上下文 → 构建 prompt → AI 生成内容
→ 创建新 segment → 返回 branch + segment
```

## ChronosMirror 升级改进点

### 1. 角色建模
当前分支创建没有角色状态管理。分叉后，不同分支中的角色可能处于完全不同的状态（存活/死亡/职位变更等），但目前系统没有跟踪这些状态。ChronosMirror 应：
- 在分支记录中保存分叉时刻的角色状态快照
- AI prompt 中应包含当前活跃角色的状态信息，确保续写内容与角色状态一致
- 分支间切换时，角色状态应自动恢复到该分支的上下文

### 2. 时间轴校验
分叉点的时间信息目前完全缺失。ChronosMirror 应：
- 每个分支记录应包含分叉的时间点（如"公元前227年"）
- 不同分支可能有不同的时间流逝速度（如军事分支时间流逝快，外交分支慢）
- 分支合并（如果支持）时需要时间轴对齐校验

### 3. MCP 维基百科
当前 AI prompt 非常简单，仅包含前文和用户方向。应通过 MCP 维基百科：
- 查询用户方向涉及的历史事件的真实细节
- 在 prompt 中注入历史背景知识，使 AI 生成的分叉内容更具历史依据
- 例如用户选择"荆轲成功刺秦王"，应查询秦王嬴政的侍卫配置、匕首的规格、大殿布局等

### 4. 节奏控制
- 分支创建的 AI 调用是同步的（非流式），用户需等待完整生成。应改为流式响应。
- 应限制分支深度（防止无限分叉导致叙事混乱）
- 可引入"分支收敛"机制——多条分支最终汇合到同一个历史节点
- 分支对话框中可添加"分叉预览"功能，让用户在确认前看到简短的剧情方向预览
