# 研究报告: src/app/api/stories/[id]/continue/route.ts

## 概述

该文件实现了故事的同步续写 API。当用户点击"续写故事"时，前端调用此 POST 接口，系统在当前分支的末尾追加一个新的故事段落。与 `stream-continue` 不同，此接口等待 AI 完整生成后才返回结果。

## 用途

提供故事的主线推进能力。AI 基于已有段落的完整上下文链，生成下一段内容，保持古典文学风格和情节连续性。

## 导出

- `callAI` 内部函数：封装 AI API 调用逻辑。
- `POST` 函数：接收 `{ branchId }` 请求体，返回新生成的段落。

## 依赖

- `@/lib/simple-db`：`storiesStore`、`segmentsStore`、`getOrderedChain`、`StorySegment`。
- AI API：使用 `AI_BASE_URL`（默认 OpenAI 兼容接口）、`AI_MODEL`（默认 gpt-3.5-turbo）、`AI_API_KEY`。

## 核心逻辑

1. 获取故事和指定分支的有序段落链。
2. 找到链尾段落（`tailSegment`）作为续写的起点。
3. 将整条链的标题+内容拼接为上下文摘要。
4. 构建 prompt，要求 AI 续写 150-300 字的古典风格内容。
5. 调用 AI API（temperature=0.7, max_tokens=2000）。
6. 创建新段落，设置 `parentSegmentId` 为尾段落、`branchId` 为当前分支。
7. 保存并返回。

## 数据流

```
POST /api/stories/[id]/continue { branchId }
→ 加载故事 → 获取有序链 → 找尾段落 → 拼接上下文
→ AI 生成 → 创建新 segment → 保存 → 返回
```

## ChronosMirror 升级改进点

### 1. 角色建模
- prompt 中完全没有角色信息。应提取当前分支中活跃的角色列表及其状态，注入 prompt。
- 角色应有"弧光"概念——续写时应考虑角色的发展方向和动机。
- 可引入角色关系图谱，AI 续写时参考角色间的恩怨关系。

### 2. 时间轴校验
- 续写没有时间推进的概念。应引入叙事时间（如"三个月后"）。
- 时间推进应与历史事件时间线对齐——如果故事设定在赤壁之战前，续写不应跳过赤壁之战。
- 应记录每个段落对应的叙事时间，便于时间轴可视化。

### 3. MCP 维基百科
- 续写时应查询维基百科获取当前时间点附近的历史事件，确保 AI 生成的内容有历史依据。
- 可在 prompt 中注入"历史事件日历"，列出当前时间前后的重要事件。
- 角色的行为应符合历史记载（如历史人物的性格特点、决策风格）。

### 4. 节奏控制
- 同步续写体验差，用户需等待完整生成。应优先使用 stream-continue。
- 可引入"续写策略"：高潮段落生成较长（300-500字），过渡段落较短（100-150字）。
- 应检测上下文长度，过长时进行摘要压缩，避免超出 AI 上下文窗口。
