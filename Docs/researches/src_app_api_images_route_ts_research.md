# 研究报告: src/app/api/images/route.ts

## 概述

`src/app/api/images/route.ts` 是古事项目中负责获取故事段落关联图片的 API 路由。该文件仅实现了一个 GET 端点，根据 `segmentId` 查询参数返回该段落对应的图片列表。

## 用途

该路由的核心用途是为故事阅读页面提供图片数据。当用户浏览故事时，前端组件 `StoryImageDisplay` 会调用此接口获取当前段落的插图、场景图、人物图等视觉内容。这是故事沉浸感的重要组成——通过图像增强用户对历史场景和人物的感知。

目前该接口处于模拟数据阶段，并未真正从数据库读取图片元数据。返回的数据结构包含 `id`、`segmentId`、`url`、`description`、`type`、`width`、`height`、`alt` 等字段。

## 导出

- `GET` 函数：处理 HTTP GET 请求，接收 `segmentId` 查询参数，返回图片数组。

## 依赖

- `next/server`：Next.js 的 `NextRequest` 和 `NextResponse` 类型，用于构建 API 路由。
- 无其他业务依赖。注意：该文件没有引入 `@/lib/simple-db`，说明图片数据层尚未与核心存储系统集成。

## 核心逻辑

1. 从请求 URL 中提取 `segmentId` 参数，若缺失则返回 400 错误。
2. 当前返回硬编码的模拟数据，包含一张 "历史场景插图"。
3. 图片类型标记为 `scene`，尺寸 800×600。
4. 异常处理：捕获错误并返回 500 状态码，附带错误详情。

## 数据流

```
前端 StoryImageDisplay → GET /api/images?segmentId=xxx → route.ts → 返回模拟图片数组
```

目前是一个完整的"空壳"——接口签名和数据结构已经定义好，但后端数据获取逻辑尚未实现。注释中明确提到"预留了从数据库获取图片的逻辑"，实际应从 `StorySegment` 的 `imageUrls` 和 `imageMetadata` 字段中获取。

## ChronosMirror 升级改进点

### 1. 角色建模
当前图片系统没有角色维度的概念。ChronosMirror 应引入角色画像系统，图片生成和展示应与角色模型绑定：
- 根据段落中出现的角色自动生成角色肖像
- 同一角色在不同段落的图片应保持视觉一致性（通过角色 embedding 或参考图实现）
- 图片类型应扩展：`character_portrait`（角色肖像）、`character_action`（角色动作）、`interaction`（角色互动）

### 2. 时间轴校验
图片生成需要时间校验：
- 确保图片中描绘的场景与段落的时间点一致（如唐代不应出现宋代服饰）
- 通过 MCP 维基百科查询历史背景，校验图片描述的时代准确性
- 图片元数据应包含 `era`、`year` 等时间标注

### 3. MCP 维基百科集成
- 当生成图片描述时，调用维基百科 MCP 获取真实的历史场景、建筑、服饰等参考信息
- 图片的 `description` 字段应由 AI 结合维基百科知识生成，而非用户手动填写
- 可引入"历史准确度评分"机制，评估生成图片与历史真实的一致性

### 4. 节奏控制
- 图片生成应有节奏控制：并非每个段落都需要配图，应根据叙事节奏智能决定
- 高潮段落（战争、决斗等）应优先配图，过渡段落可省略
- 图片加载应采用懒加载和渐进式显示，避免影响阅读流畅度
- 当前 `StoryImageDisplay` 的加载状态处理较为基础，可改进为骨架屏 + 渐进式图片解码
